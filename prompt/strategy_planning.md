# 전략 컨설팅식 기획 방법론

# **Full Execution Guideline**

(Commands + Checklists)

---

# 1. 문제 정의 (Problem Definition)

## ✅ **1단계: 실행 지시문 (Command)**

1. 해결하려는 주제를 한 문장의 질문으로 작성하라.

   * 예: “왜 GLP-1 사용자들은 3개월 후에 이탈하는가?”

2. 이 질문이 너무 넓지 않은지 검사하라.

   * “누가, 언제, 무엇을 하려는 상황에서”를 포함해라.

3. 성공을 판단할 정량 지표를 최소 1개 이상 정의하라.

   * 예: “유지율 20% 상승”, “임대율 95% 달성”

4. 프로젝트 범위(Scope)를 시간·대상·영향 기준으로 명확히 한 줄씩 작성하라.

5. 문제를 ‘현상’이 아닌 ‘근본 문제’로 재정의하라. (Why?를 3번 반복)

---

## ✅ **1단계: 체크리스트 (Checklist)**

* [ ] 문제 정의가 문장형 질문인가? (명사 나열 금지)
* [ ] 누가(Who) / 언제(When) / 무엇(What)을 포함했는가?
* [ ] 성공 지표가 숫자로 정의되어 있는가?
* [ ] Scope가 명확하게 제한되었는가?
* [ ] 현상이 아니라 해결 가능한 문제로 재정의했는가?
* [ ] 경영진에게 “이 질문만 해결하면 충분하다”라고 말할 수 있는가?

---

# 2. 구조화 (Structuring)

## ✅ **2단계: 실행 지시문 (Command)**

1. 핵심 질문을 3~5개의 세부 질문으로 분해하라. (Issue Tree 1차 분해)
2. 각 세부 질문을 다시 2~3개씩 더 쪼개라. (2차 분해)
3. 모든 분해 항목이 **MECE(중복 없고, 빠짐 없음)**인지 검증하라.
4. “만약 이 질문이 참/거짓이면 어떤 의미인가?”를 모두 기록하라.
5. 가장 중요한 질문 1~2개(결정변수)를 선정하라.

---

## ✅ **2단계: 체크리스트**

* [ ] Issue Tree가 ‘왜?’ 또는 ‘어떻게?’ 형태로 논리적으로 전개되었는가?
* [ ] 항목 간 중복은 없는가?
* [ ] 중요한 요인이 빠지지 않았는가?
* [ ] 전체 질문을 단 두세 개의 핵심 질문으로 집약할 수 있는가?
* [ ] “이 질문의 답만 알아도 80% 해결된다”라고 말할 수 있는 가지가 있는가?

---

# 3. 가설 수립 (Hypothesis)

## ✅ **3단계: 실행 지시문 (Command)**

1. Issue Tree의 각 가지마다 ‘가설’을 문장으로 작성하라.

   * 예: “초기 부작용 관리 실패가 주요 이탈 원인이다.”

2. 각 가설을 증명하기 위해 필요한 **Proof Points**를 3개씩 적어라.

3. 가설별로 검증 방법을 지정하라.

   * 인터뷰, 데이터, 로그 분석, 시장 조사 등

4. 가설을 Impact(영향도) × Confidence(확신도)로 점수화하라.

5. Impact는 크지만 Confidence가 낮은 가설을 우선 검증대상으로 설정하라.

---

## ✅ **3단계: 체크리스트**

* [ ] 모든 이슈에 대해 ‘가설 → 근거’ 형태로 연결되어 있는가?
* [ ] 가설은 팩트가 아니라 “주장”으로 명확히 적었는가?
* [ ] Proof Point(필요 근거)가 구체적인가?
* [ ] 검증 계획이 실행 가능하고 일정 내 수행 가능한가?
* [ ] 주요 가설이 Impact 중심으로 우선순위화 되었는가?

---

# 4. 분석 (Analysis)

## ✅ **4단계: 실행 지시문 (Command)**

1. 정량/정성 분석 할 일을 각각 나열하라.

2. 정량은 데이터 테이블을 먼저 만들고 필요한 행/열을 정의하라.

3. 정성은 인터뷰, 사용자 여정, 벤치마킹 자료를 먼저 수집하라.

4. 분석하는 모든 항목에 대해 세 가지를 반드시 기록하라:

   * *Insight (무엇을 발견했는가)*
   * *Implication (무슨 의미인가)*
   * *Risk (남은 불확실성은 무엇인가)*

5. 가설을 지지하는 근거와 반박 근거를 모두 정리하라.

6. 분석 결과가 Issue Tree의 구조에 정확히 매핑되도록 정렬하라.

---

## ✅ **4단계: 체크리스트**

* [ ] 정량 분석과 정성 분석이 균형 있게 수행되었는가?
* [ ] 분석 결과가 가설을 강화하거나 수정하는 역할을 했는가?
* [ ] Insight/Implication/Risk가 각각 명확히 적혀 있는가?
* [ ] 데이터가 ‘예쁜 그래프’가 아니라 ‘의사결정’으로 이어지는가?
* [ ] 결과가 Issue Tree 구조에 완벽히 연결되어 있는가?

---

# 5. 스토리라인 제작 (Storylining)

## ✅ **5단계: 실행 지시문 (Command)**

1. 가장 먼저 “결론”부터 한 문장으로 작성하라.

   * 예: “사용자의 이탈의 62%는 부작용 관리 실패 때문이다.”

2. 결론을 지지하는 3개의 핵심 메시지(Pillars)를 선정하라.

3. 각 메시지 아래 Supporting Evidence를 2~3개씩 배치하라.

4. 슬라이드 한 장당 한 메시지 원칙으로 배치하라.

5. Barbara Minto의 구조(SCQA 또는 Pyramid Principle)로 전체 스토리라인을 만든다.

---

## ✅ **5단계: 체크리스트**

* [ ] 결론 → 근거 → 데이터 순서로 논리적 흐름이 있는가?
* [ ] 한 페이지 한 메시지 원칙을 따르는가?
* [ ] Pillars 3개가 MECE히 구성되었는가?
* [ ] 불필요한 자료, 미완성 자료는 제거했는가?
* [ ] 처음 보는 경영진도 60초 만에 전체 스토리를 이해할 수 있는가?

---

# 6. 솔루션 설계 (Solution Design)

## ✅ **6단계: 실행 지시문 (Command)**

1. 최소 3개의 전략 옵션을 생성하라.

   * Baseline
   * Moderate
   * Aggressive

2. 각각의 옵션에 대해

   * Impact
   * Cost
   * Feasibility
   * Risk
     를 점수화 하라.

3. Impact vs Effort 매트릭스로 우선순위를 도출하라.

4. 각 옵션의 ‘전제 조건’과 ‘리스크’를 문서화하라.

---

## ✅ **6단계: 체크리스트**

* [ ] 최소 3개의 옵션이 제시되었는가?
* [ ] 옵션 간 수준(Level)이 다르게 설계되었는가?
* [ ] Impact vs Effort가 적절히 구분되었는가?
* [ ] 실행 리스크가 현실적으로 정의되었는가?
* [ ] “왜 이 옵션을 선택해야 하는지” 논리적·수치적으로 설명 가능한가?

---

# 7. 실행 로드맵 (Execution Roadmap)

## ✅ **7단계: 실행 지시문 (Command)**

1. 30-60-90 구조로 Action Plan을 작성하라.
2. 각 단계당 Owner(담당자), Output, Deadline을 명확화하라.
3. KPI를 정의하고 주간/월간 모니터링 플랜을 만든다.
4. 리스크 발생 시 대응 플랜(Contingency Plan)을 별도로 작성한다.

---

## ✅ **7단계: 체크리스트**

* [ ] 30-60-90 일정이 현실적인가?
* [ ] Owner/Output/Deadline이 구체적인가?
* [ ] KPI가 숫자로 표현되었는가?
* [ ] 리스크 대응 방안이 존재하는가?
* [ ] 투자자 또는 경영진이 보더라도 “실행 가능하다”고 판단할 수 있는가?

---

# 8. 경영진 커뮤니케이션 (Executive Communication)

## ✅ **8단계: 실행 지시문 (Command)**

1. 초반 30초에 결론을 먼저 말하라.
2. 이후 3개의 구조화된 메시지를 제시하라.
3. 모든 슬라이드는 “그래서, 무엇을 해야 하는가?”로 끝나야 한다.
4. 숫자 중심, 논리 중심으로 5분 이내 발표 가능하게 만든다.
5. 예상되는 10개의 질문과 답변(Q&A)을 준비한다.

---

## ✅ **8단계: 체크리스트**

* [ ] 결론을 가장 먼저 제시했는가?
* [ ] 3개 메시지로 구성되었는가?
* [ ] 모든 요소가 의사결정과 연결되는가?
* [ ] 발표 시간 5분 이내로 압축되었는가?
* [ ] 예상 질문에 대응 가능한가?
