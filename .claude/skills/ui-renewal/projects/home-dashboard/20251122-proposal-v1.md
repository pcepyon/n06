# Home Dashboard Screen Improvement Proposal

## Overview
Redesign the Home Dashboard Screen to align with the Gabium Design System, focusing on brand consistency, visual hierarchy, data readability, and motivation-centric design. The dashboard currently uses inconsistent colors (blue/orange/purple), lacks proper spacing, and has low visual hierarchy. This proposal will transform it into a cohesive, motivating experience that aligns with Gabium's "soft friendliness" and "trust through stability" principles.

## Current State Issues

### Brand Consistency Issues
- **Inconsistent Colors**: Uses blue[50], orange[50], purple[50] backgrounds instead of Gabium Primary/Neutral palette
- **AppBar**: Default Material AppBar style, doesn't use Gabium Design System
- **Typography**: Hardcoded font sizes (24px, 18px, 12px) instead of Design System type scale (3xl, lg, xs)
- **Border Radius**: Uses 12px radius on cards inconsistently (Design System defines sm=8px, md=12px, lg=16px)
- **Progress Colors**: Uses generic blue/green instead of Primary (#4ADE80)
- **Shadows**: No shadows on cards (Design System requires sm/md shadow for elevation)

### Visual Quality Issues
- **Weak Hierarchy**: Section titles (18px bold) are not distinct enough from content
- **Inconsistent Spacing**: Uses 16px, 24px, 32px padding without following Design System spacing scale
- **Card Backgrounds**: Colored backgrounds (blue[50], orange[50], purple[50]) feel too playful for medical app
- **Icon Sizing**: Hardcoded icon sizes (20px, 28px, 32px) don't use Design System icon scale
- **Timeline Connector**: 2px gray line is too thin and weak visually
- **Badge Layout**: Wrap layout causes unpredictable visual flow, lacks grid alignment

### UX Issues
- **No Bottom Navigation**: Users must use back button or AppBar actions to navigate between main features (/home, /tracking/weight, /tracking/symptom, /dose-schedule, /coping-guide, /settings)
- **Quick Action Buttons**: 2x2 grid of circular buttons feels cramped, unclear tap targets (GestureDetector wraps entire column, not 44x44px min)
- **Progress Item**: Percentage text below progress bar is redundant with fraction display (3/7)
- **Empty States**: Grey box with "아직 뱃지가 없습니다" lacks motivational encouragement
- **GreetingWidget**: Insight message uses hardcoded blue color, not Primary
- **Refresh Interaction**: RefreshIndicator present but not visually indicated (color, animation)
- **Error State**: Generic error icon/message lacks actionable guidance
- **Loading State**: Default CircularProgressIndicator, not branded

### Navigation Architecture Issue
- **Current**: Single screen with AppBar actions (only settings button)
- **Problem**: Core features (/tracking/weight, /tracking/symptom, /dose-schedule, /coping-guide) are hidden in Quick Action buttons, requiring 1-2 taps
- **Opportunity**: Introduce Bottom Navigation Bar to surface main features, reduce navigation depth

## Improvement Direction

### Change 1: Introduce Bottom Navigation Bar
- **Current:** No bottom navigation, only AppBar with settings button
- **Proposed:** 5-tab Bottom Navigation Bar: 홈(Home), 기록(Tracking), 일정(Schedule), 가이드(Guide), 설정(Settings)
- **Rationale:**
  - Surfaces core features for immediate access (current Quick Action requires taps)
  - Industry standard for health apps (Noom, MyFitnessPal, Fitbit)
  - Reduces navigation depth from Home → Quick Action → Feature to single tap
  - Persistent navigation aids discoverability and reduces cognitive load
  - Aligns with mobile-first design principles
- **Design System Mapping:**
  - Component: Section 7 - Layout Patterns - Bottom Navigation
  - Height: 56px + safe area inset
  - Background: #FFFFFF (White)
  - Border Top: 1px solid #E2E8F0 (Neutral-200)
  - Shadow: Reverse md (0 -4px 8px rgba(15, 23, 42, 0.08))
  - Items: 5 tabs, evenly distributed
  - Icon: 24x24px, Active: #4ADE80 (Primary), Inactive: #64748B (Neutral-500)
  - Label: 12px (xs), Medium (500), Active: #4ADE80, Inactive: #64748B
  - Padding: 8px top, 4px bottom
- **Tab Mapping:**
  - Tab 1: 홈 → /home (current screen)
  - Tab 2: 기록 → /tracking/weight (default) with sub-menu for /tracking/symptom
  - Tab 3: 일정 → /dose-schedule
  - Tab 4: 가이드 → /coping-guide
  - Tab 5: 설정 → /settings
- **Impact:** Quick Action Widget becomes redundant (can be removed or simplified to 2 key actions)

### Change 2: Redesign GreetingWidget with Gabium Brand Colors
- **Current:** Blue[50] background (#E3F2FD), hardcoded blue text for insight message
- **Proposed:** Neutral-50 background (#F8FAFC), Primary (#4ADE80) accent for insight message, refine typography hierarchy
- **Rationale:**
  - Blue[50] is not in Design System (inconsistent brand)
  - Medical app needs trust → Neutral palette preferred over playful colors
  - Primary green creates positive emotional response aligned with "motivation" value
  - Improved readability with proper type scale
- **Design System Mapping:**
  - Component: Card Container (Section 7)
  - Background: #FFFFFF (White, not Neutral-50 for cleaner look)
  - Border: 1px solid #E2E8F0 (Neutral-200)
  - Border Radius: md (12px)
  - Shadow: sm (0 2px 4px rgba(15, 23, 42, 0.06))
  - Padding: lg (24px) for comfortable breathing room
  - Title: 2xl (24px, Bold), Neutral-800 (#1E293B)
  - Subtitle Labels: xs (12px, Regular), Neutral-500 (#64748B)
  - Subtitle Values: lg (18px, Semibold), Neutral-800 (#1E293B)
  - Insight Message: sm (14px, Medium), Primary (#4ADE80)
  - Spacing: md (16px) between rows, lg (24px) for insight message top

### Change 3: Unify Weekly Progress Widget with Primary Color
- **Current:** Green[50] background for completed items, Blue progress indicator
- **Proposed:** Neutral-50 background for all items, Primary (#4ADE80) progress indicator, Success (#10B981) for 100% completion
- **Rationale:**
  - Consistent with Design System semantic colors (Success for achievement)
  - Reduces visual noise (3 items with green/grey backgrounds)
  - Primary color reinforces brand while indicating progress
  - Clear visual distinction between in-progress (Primary) vs completed (Success)
- **Design System Mapping:**
  - Container Background: Transparent (no background wrapper)
  - Section Title: lg (18px, Semibold), Neutral-800 (#1E293B)
  - Progress Item Background: #F8FAFC (Neutral-50)
  - Progress Item Border: 1px solid #E2E8F0 (Neutral-200)
  - Progress Item Border Radius: sm (8px)
  - Progress Item Padding: md (16px)
  - Label: base (16px, Medium), Neutral-700 (#334155)
  - Fraction: sm (14px, Regular), Neutral-500 (#64748B)
  - Progress Bar Height: 8px (thick variant from Design System)
  - Progress Bar Background: #E2E8F0 (Neutral-200)
  - Progress Bar Fill (In-Progress): #4ADE80 (Primary)
  - Progress Bar Fill (Completed): #10B981 (Success)
  - Progress Bar Border Radius: full (999px)
  - Percentage: sm (14px, Medium), color matches fill (Primary or Success)

### Change 4: Simplify Quick Action Widget (If Bottom Nav Introduced)
- **Current:** 2x2 grid with 4 circular buttons (체중 기록, 부작용 기록, 투여 기록, 대처 가이드)
- **Proposed:** Remove entirely OR reduce to 2 contextual quick actions (e.g., "오늘 체중 기록", "오늘 증상 기록")
- **Rationale:**
  - Bottom Nav surfaces all 4 actions already (redundant)
  - If kept, focus on time-sensitive actions (today's tracking)
  - Simplifies screen, reduces visual clutter
  - Current GestureDetector lacks proper touch feedback (should use GabiumButton)
- **Design System Mapping (if kept):**
  - Component: GabiumButton - Tertiary variant (from Design System Section 7)
  - Background: #F1F5F9 (Neutral-100)
  - Text: #334155 (Neutral-700), Semibold (600)
  - Border: None
  - Border Radius: sm (8px)
  - Height: 52px (Large button for easy tapping)
  - Padding: md (16px) horizontal
  - Icon: 24px, left-aligned, 12px spacing to text
  - Hover: Background #E2E8F0 (Neutral-200)
  - Active: Background #CBD5E1 (Neutral-300)
  - Layout: Column with 2 full-width buttons, md (16px) spacing

## Bottom Navigation Recommendation

### Recommendation: ✅ **Strongly Recommend Introducing Bottom Navigation Bar**

### Rationale:

**1. User Flow Analysis:**
- Current flow: Home → Quick Action → Feature (2 taps)
- Proposed flow: Tap Bottom Nav tab (1 tap)
- **50% reduction in navigation depth** for core features

**2. Industry Standard:**
- Health apps (Noom, MyFitnessPal, Fitbit) all use Bottom Nav
- Users expect persistent navigation for daily-use features
- GLP-1 treatment requires daily tracking → Bottom Nav reduces friction

**3. Feature Accessibility:**
- Current: 4 features hidden in Quick Action buttons
- Current: Settings, Profile Edit, Data Sharing require AppBar or in-screen links
- Proposed: 5 main features always visible and 1-tap accessible
- Improved discoverability for new users

**4. Cognitive Load:**
- Current: Users must remember where features are located
- Proposed: Visual menu always present (icon + label)
- Reduces mental effort, especially for older users (40-50+ demographic)

**5. Design System Alignment:**
- Design System Section 7 already defines Bottom Navigation specification
- Aligns with Gabium's "Efficiency" principle (직관적으로 시각화)
- Consistent with mobile-first best practices

**6. Implementation Effort:**
- Medium effort: Create GabiumBottomNavigation component (~1 file)
- Medium effort: Refactor router with ShellRoute (~1 file modification)
- Medium effort: Update 6 main screens to include Bottom Nav (~6 file modifications)
- High value: Permanent improvement for all future features

**7. Scalability:**
- Current Quick Action Widget doesn't scale beyond 4 features
- Bottom Nav supports 5 tabs comfortably (industry max recommended)
- Future features can be nested under existing tabs (e.g., sub-menu for "기록")

**8. Trade-offs:**
- **Pros:** Faster navigation, better discoverability, industry standard, future-proof
- **Cons:** Slightly reduces vertical space (56px), requires router refactor, all screens need Bottom Nav integration
- **Mitigation:** 56px loss is minimal on modern phones (typically 700-900px height), Quick Action removal saves ~200px

### Alternative (if Bottom Nav is rejected):
- Keep Quick Action Widget, redesign with Tertiary buttons (GabiumButton)
- Add Bottom Nav later when app grows beyond 5 core features
- Accept 2-tap navigation as acceptable for MVP phase
