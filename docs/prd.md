## [GLP-1 치료 관리 MVP] Product Requirements Document (PRD)

---

### I. 제품 개요 (Product Overview)

| 항목 | 내용 |
| :--- | :--- |
| **제품명** | 비우당 (GLP-1 치료 관리 MVP) |
| **목표** | GLP-1 사용자의 복잡한 **개인별 투여 스케줄**을 자동 관리하고, **주요 증상 및 체중 변화 기록**을 데이터 기반 보고서로 자동 생성하여, 환자의 치료 안전성과 의료진과의 커뮤니케이션 효율을 극대화하며, **개인화된 인사이트와 성취 기록을 통해 사용자의 의욕을 촉진하고 지속 가능한 건강한 삶을 영위하도록 돕는다.** |
| **비전** | GLP-1 치료 과정의 불확실성을 해소하고, 사용자가 안전하고 효율적으로 목표에 도달할 수 있도록 돕는 필수적인 디지털 파트너가 되는 동시에, **긍정적인 피드백을 통해 치료 여정을 완주하도록 돕는 동기 부여 엔진**이 된다. |
| **핵심 가치** | 아래 표 참조 |
| **기술 스택** | Flutter + Supabase (BaaS) + Repository Pattern |

#### 핵심 가치 (Core Values)

GLP-1 주사제 사용자들은 주사에 대한 **거부감과 두려움**, 부작용으로 인한 **불안**, 과거 체중 감량 실패로 인한 **낮은 자존감**, 장기 치료에 대한 **의욕 유지 어려움**을 겪는다. 비우당은 이러한 감정적 어려움을 해소하며, **사용자가 스스로 변화를 느끼고 해내고 있다는 확신을 갖게** 한다.

| 핵심 가치 | 사용자가 느끼는 것 | 사용자 관점 메시지 | 구현 방식 |
| :--- | :--- | :--- | :--- |
| **자기 확인** | 내가 해내고 있다는 확신 | "12주 후 달라진 나를 만나보세요" | 변화 시각화, 과거 기록 비교, 진행도 그래프 |
| **안심감** | 불편해도 괜찮다는 확신 | "겁먹지 마세요, 90%가 2주 내 좋아져요" | 증상 정상화 데이터, 즉각적 대처법 제공 |
| **성취 실감** | 작은 변화가 쌓이는 느낌 | "작은 변화가 쌓여 달라진 나를 만나요" | 마일스톤 축하, 뱃지, 성취 애니메이션 |
| **부담 해소** | 복잡한 건 신경 안 써도 된다는 편안함 | "치료에만 집중하세요. 나머진 맡겨두세요" | 자동 스케줄, 알림, 리포트 자동 생성 |
| **안전감** | 혼자 걱정하지 않아도 된다는 믿음 | "위험할 땐 바로 알려드릴게요" | 응급 증상 감지, 전문가 상담 연계 |

### II. Stakeholders

| Stakeholder | 역할 및 핵심 책임 | MVP 관점에서의 기대 사항 |
| :--- | :--- | :--- |
| **CEO/Product Owner** | 제품 비전 및 출시 결정권자 | 핵심 시나리오(SC0-SC4)의 오류 없는 구현과 알파 런칭 일정 준수. |
| **CTO (본인)** | 아키텍처 설계, 개발 총괄 및 리스크 관리 | 간결하고 확장성 있는 구조로 신속한 개발 iteration 달성 및 기술 부채 최소화. |
| **AI Agent/Development Team** | Flutter/BaaS 기반의 명세 구현 | 명확한 기능 상세 명세(F000-F006)를 기반으로 신속하고 안정적인 코드 작성. |
| **Target User (GLP-1 사용자)** | 앱 사용 및 피드백 제공 | 간편한 기록 관리 외에, **지속적인 성취 확인과 긍정적인 메시지를 통해 꾸준히 치료 목표를 달성할 수 있는 동기 부여.** |
| **Medical Professional** | 환자 데이터 기반의 상담 진행 | F003 (데이터 공유 모드)를 통한 명확하고 구조화된 환자 기록 열람. |

#### Target User의 감정적 프로파일

GLP-1 치료를 시작하는 사용자들이 겪는 감정적 상태와 앱이 제공해야 하는 대응을 정의한다.

| 감정 상태 | 원인 | 심리적 영향 | 앱의 대응 전략 |
| :--- | :--- | :--- | :--- |
| **거부감/두려움** | 자가 주사라는 낯선 경험 | 치료 시작 지연, 회피 행동 | 온보딩에서 점진적 교육, "처음엔 누구나 두려워요" 정상화 메시지 |
| **불안** | 메스꺼움, 변비, 구토 등 부작용 경험 | 치료 중단 고려, 약물 불신 | 부작용을 "몸이 적응하는 과정"으로 리프레이밍, 즉각적 대처 가이드 제공 |
| **낮은 자존감** | 과거 다이어트 실패, 요요 경험 | "이번에도 실패할 것" 예상, 동기 저하 | 타인과 비교 없는 개인화 피드백, 작은 성공 축하, 실패 없는 기록 (쉬어가는 날) |
| **의욕 저하** | 6개월 이상 장기 치료의 지루함 | 기록 중단, 앱 이탈 | 마일스톤 뱃지, 여정 타임라인으로 진행 시각화, 따뜻한 복귀 메시지 |
| **고립감** | 주변에 같은 치료를 받는 사람 부재 | 정보 부족, 외로움 | 치료 여정 타임라인으로 개인의 역사 강조, "함께 걷고 있어요" 연결감 메시지 |

### III. 포함 페이지 (Included Pages - P0 Scope)

MVP는 최소한의 기능을 구현하는 것을 목표로 하며, 주요 기능(F-001 ~ F006) 진입에 필요한 페이지만 포함한다.

1.  **로그인/가입 화면 (F-001):** 카카오/네이버 소셜 로그인, 이메일/비밀번호 로그인, 비밀번호 재설정
2.  **온보딩/목표 설정 화면 (F000):** 초기 프로필 및 투여 계획 설정
3.  **홈 대시보드 (F006):** 치료 진행 상황 요약, 퀵 액션 (체중, 증상, 투여)
4.  **기록 화면:**
    *   **체중/증상 기록 화면 (F002):** 간편 입력, F004 (대처 가이드) 연동
    *   **투여 스케줄러 화면 (F001):** 스케줄 관리, 투여 기록, 부위 관리
5.  **안전 관리 화면:**
    *   **증상 체크 화면 (F005):** 심각 증상 체크리스트, 전문가 상담 권장
6.  **데이터 공유 모드 (F003):** 읽기 전용 요약 리포트 전용 화면
7.  **설정 화면:** 프로필/목표 수정, 투여 계획 수정, 로그아웃

### IV. 사용자 여정 (User Journey)

각 시나리오는 **기능적 목표**와 함께 **감정적 목표**를 달성해야 한다.

| 시나리오 No. | 타겟 유저 Segment | 여정 단계 (페이지 명시) | 핵심 기능 | 감정적 목표 (사용자 관점) |
| :--- | :--- | :--- | :--- | :--- |
| **SC0. 치료 여정 시작** | GLP-1 처방 후 앱을 처음 사용하는 신규 사용자 | 로그인/가입(F-001) → 온보딩/목표 설정(F000) → 홈 대시보드(F006) | F-001, F000 | **두려움 → 희망**: "이번엔 다를 수 있어요. 의지가 아닌 과학이 도와줄 거예요" |
| **SC1. 일상 투여 및 기록** | GLP-1 치료 중 메스꺼움 등 부작용을 겪은 사용자 | (시스템 푸시 알림) → 투여 스케줄러(F001) → 체중/증상 기록(F002) → 대처 가이드(F004) → 홈 대시보드(F006) | F001, F002, F004 | **불안 → 안심**: "겁먹지 마세요, 90%가 2주 내 좋아져요" |
| **SC2. 전문가 상담** | 정기 상담을 앞두고 데이터를 공유해야 하는 사용자 | 홈 대시보드(F006) → 데이터 공유 모드 진입(F003) → (상담 진행) → 공유 종료 → 홈 대시보드(F006) | F003 | **무력감 → 주도권**: "진료실에서 말문이 막혀도 괜찮아요" |
| **SC3. 심각한 증상 대응** | 구토 등 심각한 증상이 24시간 지속되어 불안감을 느끼는 사용자 | 홈 대시보드(F006) → 증상 체크(F005) → (전문가 상담 필요 안내) | F005 | **공포 → 안전감**: "혼자 걱정하지 않아도 돼요. 위험할 땐 바로 알려드릴게요" |
| **SC4. 일상 동기 부여** | 치료 목표를 꾸준히 달성하고자 하는 사용자 | 홈 대시보드(F006) 접속 → 개인화 인사/뱃지/주간 진행도/리포트/타임라인 확인 | F006 | **고립 → 확신**: "내가 해내고 있다는 걸, 눈으로 확인해요" |
| **SC5. 기록 공백 후 복귀** | 며칠간 기록을 하지 않고 앱에 복귀한 사용자 | 홈 대시보드(F006) 접속 → 따뜻한 복귀 메시지 확인 → 간편 기록 진입 | F006, F002 | **죄책감 → 수용**: "쉬어가는 것도 괜찮아요. 다시 시작하면 돼요" |

### V. IA (Information Architecture)

MVP의 신속한 개발과 간결한 구조를 위해, 복잡한 탭 구조 대신 핵심 기능 위주로 트리를 구성한다. **핵심은 홈(F006)을 통한 빠른 기록 진입 및 동기 부여 요소 확인이다.**

```mermaid
graph TD
    A[App 시작] --> B(로그인/가입 F-001)
    B --> C(온보딩/목표 설정 F000)
    C --> D[홈 대시보드 F006 (동기 부여 중심)]
    D --> D1[퀵 액션: 체중 기록 F002]
    D --> D2[퀵 액션: 증상 기록 F002]
    D --> D3[퀵 액션: 투여 완료 F001]
    D --> D4[투여 스케줄러 F001]
    D --> D5[트래킹/기록 관리]
    D --> D6[안전/가이드]
    D --> D7[데이터 공유 모드 F003]
    D --> D8[설정]

    D4 --> D4a(주사 부위 순환 관리)

    D5 --> D5a(체중/부작용 상세 이력 F002)
    D5 --> D5b(투여 기록 상세 F001)

    D6 --> D6a(증상 체크 F005)
    D6 --> D6b(부작용 대처 가이드 상세 F004)

    D8 --> D8a(프로필/목표 수정 F000)
    D8 --> D8b(투여 계획 수정 F001)
    D8 --> D8c(로그아웃)
```

### VI. 감정적 UX 원칙 (Emotional UX Principles)

모든 기능 구현 시 아래 원칙을 적용하여 일관된 감정적 경험을 제공한다.

#### 1. 언어 프레이밍 원칙

| 사용 금지 | 사용 권장 | 의도 |
| :--- | :--- | :--- |
| 부작용 | 몸의 신호 / 적응 반응 | 문제 → 피드백으로 재해석 |
| 실패 | 쉬어가는 날 | 죄책감 제거, 지속 동기 유지 |
| 아직 ~못함 | 이미 ~했음 | 부족 강조 → 성취 강조 |
| ~해야 합니다 | ~해볼까요? | 명령 → 제안으로 톤 완화 |
| 체중 감량 | 건강한 변화 | 외모 중심 → 건강 중심 |
| 경고 | 알림 / 확인 | 공포 유발 → 정보 제공 |
| 비우당이 ~해드려요 | ~하면 돼요 / ~할 수 있어요 | 공급자 → 사용자 주어 전환 |
| ~을 제공합니다 | ~을 경험하게 돼요 | 기능 설명 → 혜택 중심 |
| ~을 도와드려요 | ~만 신경 쓰면 돼요 | 수동적 수혜 → 능동적 행동 |

#### 2. 시각적 피드백 원칙

| 상황 | 시각적 반응 | 의도 |
| :--- | :--- | :--- |
| 80% 이상 진행 | 스파클 애니메이션 + 축하 아이콘 | 목표 근접 시 기대감 고조 |
| 100% 달성 | 성공 색상(녹색) + "완료!" 텍스트 | 즉각적 성취 확인 |
| 마일스톤 도달 (7일, 14일, 30일) | 뱃지 부여 + 타임라인 기록 | 장기 여정의 이정표 제공 |
| 기록 완료 | 부드러운 체크 애니메이션 | 작은 행동도 인정 |
| 복귀 사용자 | 따뜻한 환영 메시지 + 소프트 컬러 | 죄책감 없는 복귀 경험 |

#### 3. 메시지 톤 원칙

- **1인칭 복수 피하기**: "우리 함께" → "당신의 여정"
- **무조건적 긍정**: 어떤 결과든 "잘하고 있어요"로 시작
- **비교 없는 피드백**: "다른 사용자 평균" 언급 금지
- **과정 칭찬**: "체중이 줄었네요" → "꾸준히 기록하고 있네요"
- **자기결정권 강조**: "해야 해요" → "선택할 수 있어요"
- **결과 약속**: "12주 치료 여정" → "12주 후 달라진 나"
- **공감 선행**: 기능/해결책 전에 Pain point 공감 먼저
- **구체적 숫자**: "곧 좋아져요" → "90%가 2주 내 좋아져요"

#### 4. 터치포인트별 감정 목표

| 터치포인트 | 사용자의 예상 감정 | 사용자가 경험하는 것 | 목표 감정 |
| :--- | :--- | :--- | :--- |
| 첫 로그인 | 기대 + 불안 | "12주 후 달라질 나를 상상하게 돼요" | 희망 |
| 부작용 기록 시 | 불안, 걱정 | "뭘 해야 하는지 바로 알 수 있어요" | 안심 |
| 체중 정체 시 | 좌절, 자책 | "다른 변화들을 확인하며 괜찮다는 걸 알게 돼요" | 인내 |
| 투여 완료 시 | 뿌듯함 | "해냈다는 느낌을 바로 받아요" | 성취 |
| 며칠 공백 후 복귀 | 죄책감 | "다시 시작하면 된다는 걸 느껴요" | 수용 |
| 마일스톤 달성 | 성취감 | "내가 여기까지 왔다는 걸 눈으로 확인해요" | 자부심 |

#### 5. 카피라이팅 원칙 (User-Centric Copy)

모든 사용자 대면 텍스트(UI, 알림, 온보딩, 랜딩 등)는 아래 원칙을 따른다:

| 원칙 | 설명 | Before | After |
| :--- | :--- | :--- | :--- |
| **주어 전환** | "앱이 ~해요" → "당신이 ~해요" | "비우당이 스케줄을 관리해요" | "다음 투여일, 잊어버려도 괜찮아요" |
| **혜택 중심** | 기능 → 사용자가 얻는 결과 | "자동 알림 기능" | "알림 오면 맞으면 돼요" |
| **공감 선행** | Pain point 공감 후 해결책 | "진료 리포트 제공" | "진료실에서 말문이 막혀도 괜찮아요" |
| **결과 약속** | 구체적 변화 제시 | "12주 치료 여정" | "12주 후, 거울 속 나를 기대해 보세요" |
| **두려움 해소** | 불안 → 괜찮다는 확신 + 근거 | "부작용은 정상이에요" | "겁먹지 마세요, 90%가 2주 내 좋아져요" |
| **행동 단순화** | 복잡함 → 간단한 다음 행동 | "용량과 스케줄을 확인하세요" | "알림만 확인하면 돼요" |

**카피 작성 체크리스트:**
- [ ] 주어가 "비우당/앱"이 아닌 "당신/나"인가?
- [ ] 기능이 아닌 사용자가 얻는 결과를 말하고 있는가?
- [ ] 공감 문구가 해결책보다 먼저 나오는가?
- [ ] 구체적 숫자나 시간이 포함되어 있는가?
- [ ] 다음에 할 행동이 단순하고 명확한가?

감정적 UX의 효과를 측정하기 위한 지표:

| 지표 | 측정 방법 | 목표 |
| :--- | :--- | :--- |
